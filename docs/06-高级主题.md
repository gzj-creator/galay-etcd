# 06-高级主题

本页分两类内容：

1. 已从源码确认的高级用法与兼容性信息
2. 明确标注为“说明性、未验证”的模式

## 1. 连接复用与直接返回结果

当前实现适合“连接建立一次、执行多次操作、最后关闭”的用法：

- 同步路径：`connect()` 后连续调用 `put/get/del/...`
- 异步路径：`co_await connect()` 后复用同一个客户端完成多次请求

但要注意：

- 每次调用都会直接返回结构化结果；建议就地消费，不要把 `std::expected` 结果跨层延后太久
- 客户端实例仍未声明为线程安全；共享同一个实例做并发操作仍不推荐

## 2. 大前缀查询的内存边界

前缀查询会直接返回 `std::vector<EtcdKeyValue>`。对于大 key 空间：

- 优先传 `limit`
- 把 key 设计成清晰前缀
- 不要把“前缀查询”当成无限制全量扫描接口

## 3. module / import 兼容性

公开 module 名称是：

```cpp
import galay.etcd;
```

但当前仓库只在受支持工具链下启用 import 编译路径。请把下面两件事分开理解：

- `GALAY_ETCD_INSTALL_MODULE_INTERFACE=ON`：安装 `.cppm` 文件
- `GALAY_ETCD_ENABLE_IMPORT_COMPILATION=ON` 且工具链满足要求：真正启用 module/import 编译

当前支持矩阵来自 `cmake/option.cmake`：

| 条件 | 要求 |
| --- | --- |
| CMake | `>= 3.28` |
| Generator | `Ninja` 或 `Visual Studio` |
| GCC 路径 | `Linux + GCC >= 14` |
| Clang 路径 | 非 `AppleClang` 且存在 `clang-scan-deps` |

## 4. 说明性模式：服务注册 / 发现

可以用“租约 + prefix 查询”实现最小服务注册思路，但当前仓库没有把这套模式做成真实 `examples/` 或 `test/`。因此：

- 这类模式只能视为设计提示
- 生产落地前请先补真实示例、故障恢复逻辑和验收测试

## 5. 说明性模式：配置中心

基于 prefix `get()` 拉取配置、基于 `pipeline()` 批量写入配置，在 API 层是可表达的；但当前仓库没有提供：

- Watch 增量订阅以 `AsyncEtcdClient::watch(...)` 形式提供，但目前只支持单 key watch
- 配置变更推送
- 鉴权 / TLS

所以它更像“配置快照读写”基础，而不是完整配置中心产品。

## 6. 说明性模式：简化版分布式锁

这一点必须明确：

- 当前公开 API 没有 compare/failure 事务构造器
- 只有“空 compare + success ops”的 pipeline

因此，文档里任何仅靠 `put(key, "locked", lease_id)` 实现的“锁”都只能算说明性草图，不应宣称为安全的生产级互斥方案。

如果你需要真正的分布式锁，至少需要：

- compare-and-swap / 条件事务
- 明确的失败分支
- 认证与传输安全
- 可验证的竞争测试

## 7. 兼容性与已知限制

| 主题 | 当前状态 | 备注 |
| --- | --- | --- |
| HTTP endpoint | 支持 | 推荐 `http://host:port` |
| HTTPS / TLS | 不支持 | 客户端会拒绝 `https://` |
| 认证 | 不支持 | 无用户名密码 / token API |
| Watch | 部分支持 | 仅 `AsyncEtcdClient` 单 key watch |
| 官方 Lock / Election | 不支持 | 无对应公开 API |
| Cluster / Member 管理 | 不支持 | 无对应公开 API |
| Pipeline compare/failure | 不支持 | 仅固定 success-ops txn |
| 线程安全共享 | 不建议 | 最近结果缓存会被覆盖 |

## 8. 企业接入建议

如果你要把该库接入企业环境，建议先补齐以下内容再扩大使用范围：

1. 传输安全（TLS）
2. 认证与凭据管理
3. 条件事务 / CAS
4. 完整的 watch 生命周期管理与更丰富的过滤能力
5. 针对本地 / CI / 压测环境的可重复验证脚本
